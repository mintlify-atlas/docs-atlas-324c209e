---
title: 'VMScript: Precompiled Sandbox Scripts'
description: 'Compile JavaScript once and run it many times across VM or NodeVM instances. VMScript caches bytecode and separates compilation from execution.'
---

`VMScript` is a compiled, reusable code container. Instead of parsing and compiling your script on every call to `vm.run()`, you compile it once into a `VMScript` and pass that object to `run()` repeatedly. This is especially useful when the same script is executed many times — for example, in a hot request path or a rules engine — because the compilation overhead is paid only once.

A `VMScript` is not bound to any specific `VM` or `NodeVM` instance. The binding happens at run time, just for that single execution. The same `VMScript` can be run against different sandbox instances.

## Constructor

`VMScript` supports two constructor signatures:

```typescript
// Signature 1: code + path string + options object
new VMScript(code: string, path: string, options?: {
  lineOffset?: number;
  columnOffset?: number;
  compiler?: "javascript" | "typescript" | "coffeescript" | CompilerFunction;
  compilerOptions?: Record<string, any>;
})

// Signature 2: code + options object (filename inside options)
new VMScript(code: string, options?: {
  filename?: string;
  lineOffset?: number;
  columnOffset?: number;
  compiler?: "javascript" | "typescript" | "coffeescript" | CompilerFunction;
  compilerOptions?: Record<string, any>;
})
```

## Options

<ParamField body="filename" type="string" default="vm.js">
  The filename shown in stack traces generated by the script. Has no effect on which file is read from disk — it is a label only. When using the two-argument form (path as the second argument), the path takes precedence.
</ParamField>

<ParamField body="lineOffset" type="number" default="0">
  Offset added to line numbers in stack traces. Useful when the script is embedded inside a larger source file and you want trace line numbers to match the original source.
</ParamField>

<ParamField body="columnOffset" type="number" default="0">
  Offset added to column numbers in stack traces, for the same purpose as `lineOffset`.
</ParamField>

<ParamField body="compiler" type="string | CompilerFunction" default="javascript">
  The compiler to apply before compiling to bytecode. Built-in values: `"javascript"`, `"typescript"`, `"coffeescript"`. A custom `(code: string, filename: string) => string` function is also accepted. The `typescript` and `coffeescript` values require the corresponding package installed in the host.
</ParamField>

<ParamField body="compilerOptions" type="Record<string, any>">
  Extra options forwarded to the compiler. The shape depends on the compiler in use.
</ParamField>

## Properties

All properties are read-only after construction.

| Property | Type | Description |
|---|---|---|
| `code` | `string` | The script source code (includes any prefix/suffix wrapping). |
| `filename` | `string` | The filename label used in stack traces. |
| `lineOffset` | `number` | The line offset applied to stack trace line numbers. |
| `columnOffset` | `number` | The column offset applied to stack trace column numbers. |
| `compiler` | `string \| CompilerFunction` | The compiler name or function specified at construction. |
| `compilerOptions` | `Record<string, any> \| undefined` | The compiler options specified at construction. |

## Methods

### `compile()`

Triggers compilation immediately. Normally, compilation happens automatically on the first `run()` call. Use `compile()` explicitly when you want to detect syntax errors at creation time rather than at run time. If called multiple times, the script is only compiled once.

Returns `this` for chaining.

```typescript
compile(): this
```

## Usage with VM

A `VMScript` run through `VM` behaves identically to passing the string directly, but avoids repeated compilation:

```javascript vmscript-with-vm.js
import { VM, VMScript } from 'vm2';

const vm = new VM();
const script = new VMScript('Math.random()');
console.log(vm.run(script));
console.log(vm.run(script)); // compiled only once, runs twice
```

## Usage with NodeVM

`VMScript` also works with `NodeVM`. The script is wrapped in a CommonJS module function before execution:

```javascript vmscript-with-nodevm.js
import { NodeVM, VMScript } from 'vm2';

const vm = new NodeVM();
const script = new VMScript('module.exports = Math.random()');
console.log(vm.run(script));
console.log(vm.run(script)); // compiled once, exports a new random number each run
```

## Explicit compile and error handling

You can compile a `VMScript` eagerly to catch syntax errors before any sandbox is involved:

```javascript vmscript-compile.js
import { VM, VMScript } from 'vm2';

let script;
try {
  script = new VMScript('Math.random()').compile();
} catch (err) {
  console.error('Failed to compile script.', err);
}

const vm = new VM();
try {
  vm.run(script);
} catch (err) {
  console.error('Failed to execute script.', err);
}
```

## Debugging with file names

Providing a real file path enables breakpoints and `debugger` statements in DevTools or VS Code. The debugger can step into code running inside the sandbox:

```javascript vmscript-debug.js
import { VM, VMScript } from 'vm2';
import { readFileSync } from 'node:fs';

const file = `${__dirname}/sandbox.js`;

// Passing the filename enables breakpoints in DevTools
const script = new VMScript(readFileSync(file), file);

new VM().run(script);
```

## Not bound to any VM

The key design property of `VMScript` is that it holds compiled bytecode but carries no reference to a sandbox context. When you call `vm.run(script)`, the compiled code is bound to that VM's context for the duration of the call and released afterwards. This means:

- The same `VMScript` can be used with different `VM` or `NodeVM` instances.
- Global state from one execution does not leak into the next through the script object itself.
- You can compile scripts at startup and distribute them to worker VMs without any coupling.

<Tip>
  Compile `VMScript` instances once at application startup and store them. For code that runs hundreds or thousands of times per second, this eliminates repeated compilation overhead.
</Tip>
